# Личный кабинет (веб)

Веб-приложение для пользователей VPN/подписок: тот же аккаунт, что и в Telegram-боте — **одна база данных**, бот и сайт — **два клиента (фронта)** к общим данным.

## Идея и границы

Пользователь **привязывает сессию сайта к своему Telegram** и видит **свои подписки** и **баланс** — ту же информацию, что доступна в боте (источник правды — PostgreSQL).

**MVP (первый релиз):**

- Авторизация через Telegram.
- Отображение подписок и ссылки/идентификаторов, согласованных с ботом.
- Отображение баланса (только просмотр).

**Не входит в MVP (планируется отдельно):**

- Пополнение баланса (платёжный провайдер на сайте).
- Покупка тарифа **с общего баланса** через сайт (та же бизнес-логика, что у бота: списание + запись подписки; нужны транзакции и идемпотентность).

## Архитектура

| Принцип | Описание |
|--------|----------|
| Один пользователь | В БД пользователь идентифицируется по **Telegram user id** (`tg_id` в таблице `users` и связанных сущностях). |
| Общие данные | Подписки (`subscriptions`), баланс (`users.balance`) — **те же таблицы**, что использует бот. |
| Два фронта | Telegram-бот и веб не дублируют «истину» в своих хранилищах: читают/пишут в **одну БД** (или в будущем — в **один HTTP API**, если вынесете слой доступа). |
| Бэкенд | Достаточно **серверной части фреймворка** (например, Route Handlers / Server Actions в **Next.js**) для проверки входа Telegram и запросов к БД. Отдельный микросервис не обязателен на старте. |

## Вход через Telegram

Рекомендуемый способ для публичного сайта — **Telegram Login Widget**: после нажатия «Войти» браузер передаёт на ваш бэкенд набор полей, включая **`hash`**. Его нужно **проверить на сервере** (HMAC-SHA256 по [официальной схеме](https://core.telegram.org/widgets/login)); без этого вход считается небезопасным.

Альтернатива на ранней стадии: одноразовый токен, выданный ботом по deep-link (`/start web_...`), обмен на сессию на сайте.

После успешной проверки создаётся **сессия** пользователя (например, защищённая cookie или JWT), привязанная к `tg_id`.

## Функциональность

### Уже в концепции MVP

- Список активных (и при необходимости истёкших) **подписок** — те же записи, что видит бот.
- **Баланс** — отображение текущего значения из БД.
- Выход из аккаунта.

### Позже

- Пополнение баланса через выбранный платёжный сервис (webhook, зачисление в `users.balance`).
- Покупка подписки **за счёт баланса** на сайте — логика как у бота: проверка суммы, списание, создание/обновление подписок и при необходимости вызов панели (Remnawave и т.д.), всё в **согласованных транзакциях**.

## Технологии (рекомендуемый стек)

| Слой | Вариант |
|------|--------|
| UI + серверные маршруты | **Next.js** (App Router) |
| БД | **PostgreSQL** — та же инстанс/схема, что у бота |
| Доступ к БД | Только с сервера (пул соединений), не из браузера |
| Сессии | Cookie + серверная валидация или подписанный JWT |

Стек можно изменить; важно сохранить **серверную** проверку Telegram и **не** светить секреты и строки подключения к БД клиенту.

---

## Безопасность

- Проверка подписи данных **Telegram Login** на сервере; токен бота хранить только в переменных окружения на сервере.
- **HTTPS** в продакшене.
- Ограничение CORS и SameSite для cookie, если используете куки.
- Для будущих платежей: вебхуки только с проверкой подписи провайдера, идемпотентная обработка.

## Переменные окружения

Пример (имена уточните под проект):

| Переменная | Назначение |
|------------|------------|
| `DATABASE_URL` | Строка подключения PostgreSQL (та же БД, что у бота, или read-replica для чтения). |
| `TELEGRAM_BOT_TOKEN` | Токен бота — для проверки `hash` Telegram Login. |
| `SESSION_SECRET` | Секрет для подписи сессии / JWT. |
| `NEXT_PUBLIC_TELEGRAM_BOT_NAME` | Имя бота **без** `@` для виджета входа (если используете виджет на клиенте). |

## Roadmap

1. MVP: вход через Telegram, просмотр подписок и баланса.
2. Пополнение баланса (интеграция платежей).
3. Покупка подписки с баланса на сайте (общая бизнес-логика с ботом).
4. *(Опционально)* единый HTTP API для бота и сайта, если вырастет нагрузка или появятся мобильные клиенты.
